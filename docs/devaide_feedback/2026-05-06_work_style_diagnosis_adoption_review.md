# DevAide導入レビュー: 社会人タイプ診断

作成日: 2026-05-06
対象プロジェクト: 社会人タイプ診断
導入版: DevAide v0.11.0-product-facing-cleanup
採用セット: Core Lite / standard-lite
実行プロファイル: codex-only

## 要約

社会人タイプ診断へのDevAide導入は、現時点では「開発速度を大きく上げる」というより、後続作業の迷子防止と仕様保護に効いている。

特に今回のように、既存プロジェクトから派生して新しい診断アプリを作るケースでは、コピー元の文脈が大量に残りやすい。DevAideを適用したことで、社会人タイプ診断としての正本、作業場所、変更してはいけない診断仕様、旧文書の扱いを明文化できた。

一方で、小規模アプリの初期段階に `Team / multi-ai-full` 相当の文書量を入れると重くなる。今回は `Core Lite / standard-lite` に落とした判断が妥当だった。

## 導入前の状態

- 投資家タイプ診断をベースに、社会人タイプ診断を新規プロジェクトとして作成した。
- アプリ本体は動作しており、20問、4軸、16タイプ、結果画面、共有カード生成まで実装済み。
- ただし、コピー元の投資家タイプ診断由来の文書、画像ブリーフ、handoff、レビュー文書が多数残っていた。
- 後続AIが古い文書を読んだ場合、投資家診断の方針へ戻してしまうリスクがあった。

## 導入で実施したこと

- `AGENTS.md` を社会人タイプ診断向けに更新。
- `WORKING_LOCATION.md` で正式作業場所を固定。
- `docs/PROJECT_PRINCIPLES.md` に目的、非目的、倫理・表現境界、壊してはいけない前提を記録。
- `docs/AI_KIT_VERSION.md` に採用版、採用セット、Optional Module判断を記録。
- `docs/EXECUTION_PROFILE.md` を `codex-only` 前提に更新。
- `docs/DECISION_LOG.md` に、作業場所、派生プロジェクト化、診断仕様固定、DevAide適用を記録。
- `docs/CHANGE_INTAKE.md` に、16枚カード制作、multi-ai-full化、公開前チェックをParkとして記録。
- `docs/LEGACY_DOCS_NOTE.md` を追加し、コピー元文書を正本として扱わないことを明記。
- `docs/handoffs/2026-05-06_devaide_apply_work_style.md` を追加。

## 効果が高かった点

### 1. 派生プロジェクトの文脈汚染を止められた

今回もっとも効果があったのは、コピー元プロジェクトの古い指示を「正本ではない」と明示できたこと。

投資家タイプ診断由来の文書には、投資法、16枚カード、画像生成ブリーフ、金融テーマ向けの注意事項が多く残っていた。これを放置すると、後続AIが社会人タイプ診断と投資家タイプ診断の仕様を混ぜる可能性が高い。

`docs/LEGACY_DOCS_NOTE.md` を作ったことで、後続AIが見るべき文書と見ない文書を分けられた。

### 2. 診断仕様の保護に効いた

社会人タイプ診断では、以下がプロダクトの核になる。

- 4軸
- 16タイプ
- 20問
- タイプコード
- 結果文の安全な表現

これらを `PROJECT_PRINCIPLES.md`、`DECISION_LOG.md`、`AGENTS.md` に分散して固定したことで、AIが軽い調整のつもりで診断仕様を崩すリスクを下げられた。

### 3. Core Liteへの縮小判断がしやすかった

DevAideの適用ガイドに、採用セットの選び方があったため、今回の状態を `Team / multi-ai-full` ではなく `Core Lite / standard-lite` と判断しやすかった。

これは重要だった。初期段階で文書を増やしすぎると、運用が読まれないものになりやすい。

### 4. 後続作業へのアップグレード条件を置けた

16枚カード制作、複数AIレビュー、公開前QAが始まったら `Team / multi-ai-full` へ上げる、という条件を明文化できた。

これにより、今は軽く、必要になったら厚くする運用にできる。

## 効果が中くらいだった点

### 1. 小規模開発の即効性は限定的

現時点ではCodex単独で実装、テスト、文書整理が完結している。したがってDevAideが直接開発速度を上げた感覚は強くない。

効果は主に以下に寄っている。

- 作業場所の固定
- 仕様変更の抑制
- 後続AIへの引き継ぎ
- コピー元文書の切り分け

短時間で1人が触るだけなら、DevAideの効果は「速度向上」より「事故防止」に近い。

### 2. 既存文書が多い場合、整理コストがかかる

コピー元の文書が多かったため、全面的に削除するのではなく、正本/非正本を分ける判断が必要だった。

ここはAIが自動判断しづらい。どこまで消してよいか、何を残すべきかはプロジェクトごとに違う。

## 改善してほしい点

### 1. 派生プロジェクト用の導入モードがほしい

今回のように「既存プロジェクトをコピーして別テーマへ作り替える」ケースでは、通常の新規導入とは違う問題が出る。

必要だった処理:

- コピー元由来の文書を検出する。
- 正本として残す文書を決める。
- 古いテーマ固有のブリーフを隔離する。
- 新プロジェクトの作業場所と原則を上書きする。

DevAideに `forked-project` または `derived-project` 用チェックリストがあるとよい。

### 2. Core Liteの標準ファイルセットをもっと明確にしてほしい

Apply文書にはセット別の説明があるが、実際に小規模アプリへ入れるときは「最低この5ファイルでよい」という強い推奨があると使いやすい。

今回の体感では、Core Liteの最小正本は以下で十分だった。

- `AGENTS.md`
- `WORKING_LOCATION.md`
- `docs/PROJECT_PRINCIPLES.md`
- `docs/DECISION_LOG.md`
- `docs/handoffs/`

必要に応じて追加:

- `docs/AI_KIT_VERSION.md`
- `docs/CHANGE_INTAKE.md`
- `docs/LEGACY_DOCS_NOTE.md`

### 3. 古い文書を「消す/隔離する/残す」の判断支援がほしい

コピー元文書を全部消すと履歴や知見が失われるが、残すと後続AIが誤読する。

今回は `docs/LEGACY_DOCS_NOTE.md` で解決したが、DevAide側に以下のテンプレートがあるとよい。

```text
Legacy docs handling:
- Source project:
- Safe to reference:
- Do not use as source of truth:
- Needs rewrite before reuse:
- Delete candidate:
```

### 4. 導入後レビューのテンプレートがあるとよい

開発側へ渡すフィードバックとして、今回のような導入レビューを自然に作れるテンプレートがあると便利。

推奨項目:

- 導入前の状態
- 採用セット
- 効果があった点
- 重かった点
- 採用しなかったmoduleと理由
- 次回改善提案
- 体感スコア

## 体感スコア

| 観点 | スコア | コメント |
|---|---:|---|
| 作業場所の固定 | 9/10 | 効果が高い。後続AIが迷いにくい。 |
| 仕様保護 | 8/10 | 診断軸・タイプコードを守るのに効く。 |
| 開発速度 | 5/10 | 初期段階では速度への即効性は限定的。 |
| 後続AIへの引き継ぎ | 8/10 | 会話圧縮や再開に強くなる。 |
| 文書量の適切さ | 7/10 | Core Liteなら許容範囲。multi-ai-fullだと重い。 |
| 派生プロジェクト対応 | 6/10 | 使えるが、専用テンプレートがあるともっとよい。 |

総合評価:

```text
現段階: 7/10
16枚カード制作や公開QAへ進む場合: 8.5/10
複数AIで分担する場合: 9/10
```

## 開発側への提案

1. `derived-project` 用の導入チェックリストを追加する。
2. Core Liteの「最小正本セット」を明文化する。
3. `LEGACY_DOCS_NOTE_TEMPLATE.md` を追加する。
4. 導入レビュー用テンプレートを追加する。
5. Optional Moduleを採用しなかった理由を書く欄を、Core Liteの標準に入れる。

## 結論

社会人タイプ診断へのDevAide導入は、現時点では「開発速度を上げる道具」というより、「後続AIが道を間違えないためのガードレール」として有効だった。

特に、既存プロジェクトから派生した新規アプリでは、コピー元の文脈が残る問題が大きい。DevAideはその整理に効く。

ただし、小規模プロジェクトでは文書量を抑えることが重要。今回のように Core Lite を基本にし、必要になったら Team / multi-ai-full へ上げる流れが、実用上いちばんバランスがよい。
