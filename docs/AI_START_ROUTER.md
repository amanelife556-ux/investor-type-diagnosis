# AI Start Router: 投資家タイプ診断

作成日: 2026-05-05
管理者: ユーザー

## 目的

- いきなり実装に入るべきか、設計や整理から始めるべきかを判断する。
- AI名ではなく、役割ベースで開始担当を決める。
- 人間が判断すべきプロダクト方針変更を、AIに丸投げしない。
- AIの記憶や前回セッションの要約に依存しすぎず、外部化された判断・禁止事項・引き継ぎを確認してから始める。

## 開始前に確認する前提

作業単位を始める前に、必要に応じて以下を確認する。

- `docs/PROJECT_PRINCIPLES.md`
- `docs/DECISION_LOG.md`
- `docs/CHANGE_INTAKE.md`
- `AGENTS.md`
- `docs/handoffs/` の最新メモ
- 今回の指示書

前回のAIの記憶や要約だけで判断しない。
圧縮で失われやすい判断理由、禁止事項、やらないことは、プロジェクト内の文書を優先する。

## 開始ルール

| 状態 | 最初に使う役割 | 目的 | 次に渡す相手 |
|---|---|---|---|
| アイデアが曖昧 | 方針整理役 | 目的、対象、やらないことを整理する | 構成整理役 / プロダクト判断役 |
| 作るものは見えているが構成がない | 構成整理役 | ファイル構成、データ構造、画面構成を整理する | 実装役 |
| 仕様が明確で実装したい | 実装役 | 既存構造に沿って最小変更で実装する | レビュー役 / QA役 |
| 既存コードを改善したい | 既存コード理解に強い実装役 | 現状把握、影響範囲確認、最小修正を行う | レビュー役 |
| 実装後に客観レビューしたい | レビュー役 | バグ、仕様逸脱、過剰実装、テスト不足を確認する | 実装役 / 統合役 |
| UIや画像など視覚確認が重要 | 視覚確認役 / QA役 | スクリーンショット、レスポンシブ、アセット表示を確認する | ビジュアル担当 / 実装役 |
| 診断仕様を変えたい | プロダクト判断役 | 仕様変更の是非を判断する | 方針整理役 / 実装役 |

## このプロジェクトでの割り当て

| 役割 | 実行主体 | 備考 |
|---|---|---|
| 方針整理役 | Codex / 人間 | 目的、やらないこと、採用判断を整理 |
| 構成整理役 | Codex | App/logic/data/assets/docs の境界を整理 |
| 実装役 | Codex / 担当AI | 変更範囲を限定して実装 |
| レビュー役 | Codex / QA担当 | バグ、仕様逸脱、過剰実装、テスト不足を確認 |
| 視覚確認役 | ビジュアル担当 / QA担当 | CSS、画像、レスポンシブ、スクショ確認 |
| QA役 | Raman / Codex | テスト、ビルド、手動確認 |
| 統合役 | Codex | 外部担当の成果を統合 |
| プロダクト判断役 | 人間 | 最終判断は人間が行う |

## 判断フロー

```text
1. 目的が曖昧なら、方針整理役から始める。
2. 目的は明確だが構成が曖昧なら、構成整理役から始める。
3. 仕様と変更範囲が明確なら、実装役から始める。
4. UI/画像が絡むなら、視覚確認役/QA役を早めに入れる。
5. 実装後は、必要に応じてレビュー役/QA役へ渡す。
6. 仕様変更、依存追加、プロダクト方針変更は、人間が判断する。
```

## 現在のおすすめ開始役割

```text
Start role: 視覚確認役 / 実装役
Reason: 未対応の主課題が、診断書ポートレートの中央寄せ/切り出し修正であるため。
First execution subject: ビジュアル担当 / Codex
Next role: QA役
```

## 注意

- AI名を固定しすぎない。
- 同じAIが複数役割を担当してもよいが、役割の切り替えを明示する。
- 実装役から始める場合でも、未確認事項があるなら実装前に止める。
- プロダクト判断役をAIに完全委任しない。
