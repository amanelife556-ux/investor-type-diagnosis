# 全体レビュー 指示書: テキスト改善の統合判断

## 担当

- 全体構成/統合レビュー: Codex / 人間

## 目的

コンテンツ担当、ロジック担当、QA担当の成果を統合し、アプリ全体として採用できる品質になっているか判断する。

個別の文章の良し悪しだけでなく、診断体験として一貫しているかを見る。

## 入力として受け取るもの

- コンテンツ担当の修正文案
- ロジック担当の整合性レビュー
- QA担当の表示確認結果
- 未解決の判断事項

## 主な確認観点

- ユーザーが「自分の投資スタイル」を理解しやすいか
- 16タイプの差が自然に出ているか
- タイプ名の世界観と現実の投資法がつながっているか
- 投資助言ではなく自己理解として成立しているか
- 診断軸、質問、結果、相性、共有文が同じ方向を向いているか
- 文章量が結果画面のデザインに収まっているか
- 今すぐ直すべきものと後回しにすべきものが分かれているか

## 採用判断

各変更は以下のいずれかに分類する。

- Adopt: 今回採用する
- Adjust: 一部調整して採用する
- Park: 今回は保留し、`docs/CHANGE_INTAKE.md` に残す
- Reject: 今回は採用しない

## 変更してよいこと

- 担当者の成果の統合
- 軽微な文言調整
- 表示崩れを避けるための軽微なCSS調整
- `docs/CHANGE_INTAKE.md` への保留案記録
- `docs/DECISION_LOG.md` への重要判断記録
- `docs/GIT_UNMANAGED_HISTORY.md` への作業履歴記録

## 変更しないこと

- 人間の確認なしに診断コンセプトを変えない
- 人間の確認なしに16タイプ構造を変えない
- 担当者間で意見が割れた内容を独断で大きく変更しない
- DevAide本体は変更しない

## 必須確認

```bash
npm test
node src/__tests__/diagnosisLogic.node-test.mjs
npm run build
```

可能ならブラウザでPC/スマホ表示も確認する。

## 成果物

以下を提出する。

- 統合レビュー結果
- 採用した変更
- 保留した変更
- ユーザー判断が必要な点
- 実行したテストと結果
- 未確認事項

## 完了条件

- テキスト改善がアプリに反映されている
- ロジック整合性に重大な問題がない
- 主要画面で表示が崩れていない
- 次回作業者が状況を追える記録が残っている
