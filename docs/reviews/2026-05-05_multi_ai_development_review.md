補足: このレビューは `v0.3.0-proposal` 時点の記録として残します。現在の採用版は `docs/AI_KIT_VERSION.md` の通り `v0.9.1-dev-aide-concept` で、再適用内容は `docs/operation_reviews/2026-05-05_operation_review.md` に記録しています。

## 結論

今回のプロジェクトには、最新版キットの `AI_KIT_VERSION.md` による採用版固定が有効です。

理由は、今回の開発ではキット自体も改善中であり、プロジェクト側が「どの版のルールを、どのステータスで使っているか」を明示しないと、次回AIが最新版キットを無条件に適用してしまうリスクがあるためです。

今回の診断アプリでは、`docs/AI_KIT_VERSION.md` を追加し、`v0.3.0-proposal` を `trial` として採用しました。

## 採用した内容

### 1. プロジェクト側のキット版固定

採用ファイル:

- `docs/AI_KIT_VERSION.md`

有効だった点:

- キット本体の最新版と、プロジェクト側の採用状態を分離できる。
- `proposal` のキットを、このプロジェクトでは `trial` として試す、と明示できる。
- 次回AIが「最新版だから全部採用」と誤解しにくい。

### 2. Git管理と版固定の分離

今回のコピー先はGitリポジトリではありません。

そのため、Git履歴の代わりに `AI_KIT_VERSION.md` が「運用契約」を明示する役割を持ちます。

ただし、Gitの代替ではありません。

今後Git管理を開始する場合は、以下もコミット対象にするべきです。

- `AGENTS.md`
- `WORKING_LOCATION.md`
- `docs/AI_KIT_VERSION.md`
- `docs/PROJECT_PRINCIPLES.md`
- `docs/handoffs/`
- `VISUAL_FIX_BRIEF.md`

### 3. セッション開始時の読み物固定

最新版キットでは、セッション開始時に `docs/AI_KIT_VERSION.md` を読む流れが追加されています。

これは今回のように、作業場所、役割分担、未対応課題が複数あるプロジェクトに有効です。

次回AIが読む順番としては、以下がよいです。

1. `docs/AI_KIT_VERSION.md`
2. `AGENTS.md`
3. `docs/PROJECT_PRINCIPLES.md`
4. `WORKING_LOCATION.md`
5. `docs/handoffs/` の最新メモ
6. 今回の指示書

## マルチAI開発として良かった点

### 役割分担が明確だった

今回の役割:

- 全体構成/統合レビュー
- ビジュアル担当
- ロジック担当
- コンテンツ担当
- QA担当

この分け方により、レビュー指摘を担当者へ割り当てやすかったです。

例:

- CSS/画像切り出し問題: ビジュアル担当
- 16タイプ到達性: ロジック担当
- 投資助言に見える表現: コンテンツ担当
- ビルド/スマホ確認: QA担当
- `main.tsx` / `App.tsx` の責務整理: 全体構成

### レビュー指摘からタスク化する流れが機能した

「誰に作業を指示する？」という問いに対し、レビュー指摘を担当別タスクへ変換できました。

これは `REVIEW_TO_TASKS_TEMPLATE.md` と相性がよいです。

### 環境不足時の代替確認が有効だった

初期状態ではnpmがなく、フルテストができませんでした。

その間、Node単体スモークテストで以下を確認しました。

- エントリ構造
- 20問/4軸バランス
- 16タイプ定義
- 共有文面
- アセット参照

後からnpmを導入し、`npm test` と `npm run build` で正式確認できました。

これは `ENVIRONMENT_FALLBACKS.md` の考え方が有効な例です。

## マルチAI開発で危なかった点

### 1. 実装が二重化した

一時的に `src/main.tsx` と `src/App.tsx` の両方にアプリ実装が存在しました。

リスク:

- ビジュアル担当がどちらのDOMに合わせるべきか迷う。
- QAが実際には使われていない画面を確認する可能性がある。
- コンテンツ修正が片方にしか入らない可能性がある。

対処:

- `src/main.tsx` は起動専用に戻した。
- 画面本体は `src/App.tsx` に集約した。

教訓:

- 複数AIが触るプロジェクトでは、入口と本体の責務を早めに固定する。

### 2. 作業場所が途中で変わった

作業場所がローカルDocuments配下から外部ボリュームへ移りました。

リスク:

- 古いコピーを編集する。
- テスト結果が別コピーのものになる。
- 他AIへの指示がズレる。

対処:

- `WORKING_LOCATION.md` を作成。
- `AGENTS.md` とREADMEから参照。

教訓:

- 作業場所が変わったら、最初に固定文書を置く。

### 3. キット改善とプロジェクト運用が混ざりやすい

今回、診断アプリを作りながらキット自体も改善しました。

リスク:

- キット最新版がそのままプロジェクトルールになってしまう。
- まだ `proposal` のルールを正式採用と誤解する。

対処:

- `docs/AI_KIT_VERSION.md` で採用版とステータスを固定。

教訓:

- キットを育てるプロジェクトでは、プロジェクト側の採用版固定が必須。

### 4. アセットQAが後追いになった

タイプ別ポートレート16枚が追加されたあと、存在チェックをスモークテストへ追加しました。

リスク:

- 1枚欠けてもビルド前に気づきにくい。
- ファイル名とタイプコードがズレる。
- 画像表示の切り出し問題がロジックレビューでは見落とされる。

対処:

- スモークテストに16タイプ画像の存在確認を追加。
- `VISUAL_FIX_BRIEF.md` で切り出し問題をビジュアル担当へ分離。

教訓:

- データIDとアセットが対応する場合、初期からアセットQAを入れる。

## 今回の開発に対する評価

### 良かった点

- 役割分担が機能した。
- ロジック、コンテンツ、ビジュアル、QAを分けたことで議論が整理された。
- スモークテストがnpm不在時の安全網になった。
- npm導入後に `npm test` と `npm run build` まで通せた。
- キット改善をプロジェクト側に即反映しつつ、採用状態を `trial` として管理できた。

### 改善したい点

- 最初から `AI_KIT_VERSION.md` を置くべきだった。
- 作業場所変更時点で、AGENTS/引き継ぎ/版固定まで一括で置くべきだった。
- 画像アセット追加時点で、アセットQAチェックを同時に追加すべきだった。
- ビジュアル担当への依頼は、最初から `VISUAL_FIX_TEMPLATE.md` 型で出すともっと速かった。

## 今後の運用提案

このプロジェクトでは、次の運用を続けるのがよいです。

1. 作業開始時は `docs/AI_KIT_VERSION.md` を読む。
2. 次に `AGENTS.md`、`docs/PROJECT_PRINCIPLES.md`、`WORKING_LOCATION.md` を読む。
3. 変更前に、変更対象と変更しない範囲を明記する。
4. 仕様変更が必要なら実装を止めて人間に確認する。
5. 変更後は必ず `npm test` と `npm run build` を実行する。
6. 区切りごとに `docs/handoffs/` を更新する。
